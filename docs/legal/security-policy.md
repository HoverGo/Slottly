## 1. Назначение документа

1.1. Настоящая Политика информационной безопасности (далее — **Политика**) определяет принципы и меры защиты информации в сервисе Commerce Booking (далее — **Сервис**).

1.2. Политика распространяется на:

- персональные данные пользователей, сотрудников и клиентов;
- коммерческую информацию компаний-клиентов;
- платёжные и учётные данные;
- техническую инфраструктуру Сервиса.

1.3. Цель Политики — обеспечение конфиденциальности, целостности и доступности информации.

---

## 2. Принципы безопасности

- **Изоляция данных (multi-tenant):** данные каждой компании логически разделены; доступ к чужой компании невозможен через штатный API.
- **Минимально необходимый доступ:** права сотрудников компании ограничиваются ролевой моделью (RBAC).
- **Защита по умолчанию:** аутентификация обязательна для операций с данными; пароли не хранятся в открытом виде.
- **Прозрачность для пользователя:** пользователь управляет своим паролем и видит только те компании, к которым имеет доступ.

## 3. Классификация информации

| Категория | Примеры | Уровень защиты |
|-----------|---------|----------------|
| Публичная | Тарифы, описание API, документация | Базовый |
| Внутренняя | Логи без ПДн, метрики, конфигурация | Повышенный |

| Строго конфиденциальная | Пароли (хеши), JWT-секреты, ключи платёжных систем | Максимальный |

## 4. Аутентификация и управление доступом

### 4.1. Пользователи Сервиса

- Регистрация по email и паролю.
- Пароли хранятся в виде криптографических хешей (bcrypt).
- Сессии реализованы через JWT (Bearer-токен) с ограниченным сроком действия.
- Смена пароля — только после подтверждения текущего пароля.

### 4.2. Доступ к компаниям

- Контекст компании (`TenantContext`) устанавливается при каждом запросе к API компании.
- Участник видит только свои компании; попытка доступа к чужой компании возвращает код `404` без раскрытия факта существования ресурса.
- Владелец компании имеет полный набор прав; сотрудники — только назначенные роли (`manage_roles`, `manage_members`, `manage_branches`, `manage_schedules`, `manage_join_requests`).

### 4.3. Администраторы платформы

- Глобальный доступ (`is_platform_admin`) предоставляется ограниченному кругу лиц.
- Назначение администраторов — через защищённую конфигурацию и API с проверкой прав.
- Действия администраторов (промокоды, просмотр агрегированной статистики) должны подлежать внутреннему аудиту.

## 5. Защита данных

### 5.1. При передаче

- В production обязательно использование HTTPS (TLS 1.2+).
- JWT передаётся только в заголовке `Authorization: Bearer`.

### 5.2. При хранении

- База данных PostgreSQL с разграничением доступа на уровне сети и учётных записей.
- Секретные ключи (`SECRET_KEY`, ключи платёжных провайдеров) хранятся в переменных окружения, не в исходном коде.
- Файл `.env` не включается в систему контроля версий.

### 5.3. Изоляция на уровне БД

- Составные внешние ключи предотвращают привязку ролей и сущностей одной компании к другой.
- Все запросы к данным компании выполняются с обязательным фильтром `company_id`.

## 6. Безопасность приложения

- Валидация входных данных на уровне Pydantic-схем.
- Централизованная обработка ошибок без раскрытия внутренних деталей системы.
- Защита от типовых уязвимостей OWASP Top 10 (инъекции, XSS в API-контексте, небезопасная десериализация) — в рамках архитектуры FastAPI + SQLAlchemy.
- Регулярное обновление зависимостей (`requirements.txt`).

---

## 7. Платежи

- Обработка платежей делегируется сертифицированным платёжным провайдерам (ЮKassa и др.).
- Данные банковских карт на серверах Оператора **не хранятся**.
- Webhook-платежей принимаются только от доверенных провайдеров с верификацией подписи (при production-интеграции).
